# jp-writing

日本語の技術記事・README・リポ内の解説 Markdown に対して、文章の規範を適用する。

## 1. 対象を選ぶ

カレントディレクトリから自動判定する。

### Zenn リポ（`articles/` が存在する）

`articles/*.md` のフロントマターを走査し、`writing-reviewed: true` が**ない**記事だけを候補にする。タイトルを一覧してユーザーに選ばせる。

### その他のリポ

`find . -name '*.md'`（`.git/`・`node_modules/` 等は除外）で Markdown を探索し、一覧してユーザーに選ばせる。

## 2. 修正する

1. 対象ファイルを読む
2. プロジェクトに `context/conventions.md` 等があれば読む
3. 後述の規範に照らして違反箇所を修正する

### やらないこと

- 著者の主張・論旨を変えない。変えるのは表現だけ
- 判断に迷う箇所（意図的な表現か規範違反か区別できない）は修正せず、報告に回す

## 3. 報告する

修正後に以下を報告する:

- 修正した箇所と理由（規範のどのルールか）
- 迷って手を付けなかった箇所（あれば）
- Zenn 記事の場合: フロントマターに `writing-reviewed: true` を追加した旨

## 4. コミット・push はしない

変更はワーキングツリーに残す。コミット/push はユーザーの指示があったときのみ。

## 執筆時の参照

pr-workflow 等の他スキルから執筆目的で参照されることがある。その場合は `context/conventions.md` 等があれば読む。なければ文体・構成の方針をユーザーに質問して止まる。規範に従って書く。

# 規範

## 文体

- ですます調と、である調の混合を許容する。
- 新規執筆時は、地の文を「ます」調、体験談や所感を「る」調にする程度の自然な混合にする。

## 整形

- 段落の区切りは空行で示す。
- 段落内では、意味の区切りで改行してよい。一文ずつ機械的に改行しない。ベタ書きもしない。改行位置は書き手の判断に委ねる。
- コード、設定ファイル、ログの断片はコードブロックで示す。
- 定義や分類の列挙は箇条書きで示してよい。定義される用語は太字にする。
- ダッシュ（`—`、`――`）を日本語の地の文で使わない。補足は括弧（）に、言い換えは読点か句点で処理する。

## 冗長の排除

- 同じ主張を言い換えて繰り返さない。
- 場面や例を示した直後に、その内容を要約し直さない。意味づけの一文だけを置く。
- 接続や評価のためだけの文（「それ自体はよいことである」など）を置かない。
- 読者が自力で補える中間段階の説明は書かない。
- 想像上の読者との問答を修辞として使わない。主張はそのまま述べる。
- 「重要なのは〜である」のような前置きで主張を予告しない。主張をそのまま書く。
- 「この状況を解決するために」のような、次に書く内容を予告するだけの文は省く。

## LLM口調の禁止

次のような言い回しは、中身のない型である。使わない。

- **予告と総括**: 「本記事では〜を扱う」「まとめると」「要するに」（直前の言い換えだけのとき）、「〜に他ならない」
- **空虚な形容**: 「不可欠」「核心的」「鍵となる」「根本的な」「多角的」「包括的」
- **空虚な動詞**: 「掘り下げる」「深掘りする」「言語化する」（何をどう書いたかを示さず終わる）
- **弱い緩和**: 「〜と言えるだろう」「〜かもしれない」（根拠なく主張を弱めている場合のみ削る。推量・仮定・読者の疑念を表す場合は残す）
- **空虚な強調**: 「非常に」「極めて」「大いに」

## 視点と語り

- 対象を指す語は具体的に選ぶ。「AI」「ツール」のような広い語でぼかさない。記事内で名前がある対象は名前で呼ぶ。
- 術語・固有名を導入したら、以後はその語で通す。曖昧語に後退しない。
- 例示では、結果の羅列や受動態ではなく、行為者を主語にした動作で書く。
- 「入社2年目のエンジニアが」のような架空の人物設定を無意味に冠しない。

## 演出の節度

全面禁止ではなく、節度の規範。

- 太字強調を多用しない。誤読を防ぐ否定や帰結など、論理の要所に限る。
- 短い決め台詞を独立段落にする演出は、記事の山場に限る。
- 「〜してはならない」より「〜するわけにはいかない」のような判断の形を選ぶ。
- 慣用表現をひねった比喩や、指す内容が一意に決まらない比喩を使わない。平易な動詞でそのまま言う。
