# テスト方針

テストは変更可能性の担保である。実行者（AI）が壊したことに自分で気づくための装置であり、注意し続ける人間を前提にしない。

保証（何が壊れてはいけないか）は Issue の保証節で人間が裁可し、テストの実装は実行者が書く。この分業を **Guarantee-Driven Development (GDD)** と呼ぶ。テストそのものが正なのではなく、裁可された保証を実行可能な形に書き下ろしたものである。TDD がテストを先に書く規律だとすれば、GDD は約束の裁可を先に行う規律であり、人間の仕事はテストを書くことから約束を裁可することへ移る。Issueドリブン期のリポはこの方式で保守する（フェーズは各リポの CLAUDE.md で宣言する）。

濃淡はリスクベースで決める: 公開API・契約面は厚く、内部実装は薄く、UI・見た目は user の手動検証に回す。

fix には回帰テストを同梱する。

外部依存は DI で受け、テストは fake で差し替える（実サービスに接続しない）。

各リポは公開面の保証を `docs/guarantees.md`（保証台帳）として持てる。自然言語の保証の箇条書き＋対応テスト（ファイル・テスト名）の表で構成し、地位は design-decisions.md と同格（実装が食い違う場合は指摘する）。載せるのは契約面の保証のみで、内部実装のテストは載せない。初版敷設の経路は2つある: 新規リポは公開準備の一環として `repo-standardize` の後・`repo-readme` の前に `guarantee-audit` skill で敷く（標準経路。README が台帳へリンクするには台帳が先に要る）、既存リポは棚卸し（同じく `guarantee-audit`）。以後は Issue の保証節と同一PRでの更新により追従する。台帳とテストの乖離は `guarantee-audit` skill による定期棚卸しで検出する。台帳は README から到達可能であるべき（リンク位置は readme-guide の規定に従う）。Gaps 節（保証すべきだがテストが無いもの）は見送り理由ごと記録し、空になったら節ごと削除する。
