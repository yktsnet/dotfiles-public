# テスト方針

テストは変更可能性の担保である。実行者（AI）が壊したことに自分で気づくための装置であり、注意し続ける人間を前提にしない。

保証（何が壊れてはいけないか）は Issue の保証節で人間が裁可し、テストの実装は実行者が書く。

濃淡はリスクベースで決める: 公開API・契約面は厚く、内部実装は薄く、UI・見た目は user の手動検証に回す。

fix には回帰テストを同梱する。

外部依存は DI で受け、テストは fake で差し替える（実サービスに接続しない）。

各リポは公開面の保証を `docs/guarantees.md`（保証台帳）として持てる。自然言語の保証の箇条書き＋対応テスト（ファイル・テスト名）の表で構成し、地位は design-decisions.md と同格（実装が食い違う場合は指摘する）。載せるのは契約面の保証のみで、内部実装のテストは載せない。初版は契約棚卸しIssue（既存テストから保証を抽出し user が裁可）で作り、以後は Issue の保証節と同一PRでの更新により追従する。
