# Module Guide

OSS モジュール型リポ（ポートフォリオ兼実用）の設計規範。新規リポの型決め・モジュール追加・配布形態やデモ方式の判断に適用する。

---

## 1. リポの型を決める

正体を「汎用部分 + 固有部分」に分離して一文で定義する。固有部分の実資産（実クライアントのテンプレート・実データ・本番パラメータ）はリポに置かない。型は3つ。

| 型 | 汎用部分 / 固有部分 | 配布 |
|---|---|---|
| **組み込み型** | 利用者のビルド・実行環境に取り込まれる package / 検証フィールドとしての自分の実プロジェクト | レジストリ公開（JS → npm、Python → PyPI）。組み込まれる以上これが必須 |
| **ツールキット型** | ドメイン語彙を含まない機械（`packages/`）/ ドメイン適用例（`examples/<domain>/`、架空データのみ） | clone リファレンス |
| **研究型** | 検証の枠組み・作法（問いと手法） / 答え（パラメータ・対応表・成績）は Env / 設定の外部注入でリポ外 | clone リファレンス |

- **clone リファレンスが既定**。レジストリ公開は versioning・後方互換・英語 doc の継続コストを伴うため、組み込み型である場合か利用者が現れた場合にのみ払う。
- 言語はドメインのエコシステムに従う。「npm が標準」「pip が標準」という決め方をしない。

## 2. 構造

```
repo/
├── packages/ (or src/)   # 汎用部分。ドメイン語彙なし
│   └── <module>/
│       ├── cli.*         # モジュールごとに CLI 入口（デモと検証を兼ねる）
│       └── demo.tape     # VHS 台本（同梱してデモを再現可能に保つ）
├── tests/                # src をミラー配置
└── examples/<domain>/    # 適用例（架空・中立データのみ）
```

- 外部依存（DB・外部 API・変換サービス・時刻）は直接 import せず引数・factory で注入する。テストは fake で回す。
- 汎用部分は組み立て側（app）を import しない。一方向依存を守れば、後のレジストリ切り出しはディレクトリ移動程度で済む。
- MVP の完成条件（DoD）は「実利用者が1日使える」×「第三者が価値を 30 秒で理解できる」から逆算し、PLAN.md に列挙する。

## 3. デモ

- **常設 URL は作らない**。モジュール = VHS GIF（`.tape` 同梱、`vhs <tape>` で再生成）、アプリ = 一発起動（`docker compose up` 等）+ 画面録画 GIF。
- 機密・エッジ・実データは映さない。データソースの再配布規約にも注意（数行の提示は de minimis）。
- 成績・実測値をデモで実績として語らない（中立パラメータのデモ結果は本番と無関係）。
- セットアップ手順は各言語の標準ツール（venv+pip / npm）を正とし、Nix は任意ショートカットとして触れるだけにする（Nix 外の利用者が大多数）。
