# README Guide

公開リポの README 作成ガイド。新規リポの README を書くとき、**種別を判定し → コアメッセージとターゲットを決め → それに即したアウトライン（H2）を組む**。この順で考える。

デプロイの詳細は `cicd-guide.md`、検証手段は `harness-guide.md` にあるため、README では重複させない。

このガイドの目的は、README を一律のテンプレートに揃えることではない。**公開する各リポの README が、それぞれ筋の通った1本の主張になっている状態**を作ることにある。筋が通っているとは、次の3点が一直線につながっていることを指す。

- **コアメッセージ** … この README を見た人に、ひとことで「どういうリポか」と思ってもらうか
- **ターゲット** … 誰に向けて書くか（開発者である前提で、習熟度・職種レイヤー・特定技術の利用者）
- **アウトライン** … 上記2点が最も伝わる H2 構成

固定の必須セクションリストは置かない。代わりに、**種別ごとの「下限（必ず満たす観点）」**だけを規定し、その上のアウトラインはリポごとに動的に決める。

---

## 0. 考える順序（このガイドの使い方）

```
種別判定（Type A / B / C）
   └─ 下限：必ず満たす観点の集合（floor・順序を持たない）   ← 静的・本ガイドが規定
        └─ コアメッセージ ＋ ターゲット                      ← リポごとに決定
             └─ アウトライン（H2 の順序・命名・厚み）          ← 動的・上記から導出
```

下限は「満たすべき観点」であって、H2 の並び順や名前を固定するものではない。コアメッセージとターゲットが決まって初めて、下限を満たす形でアウトラインが決まる。

## 1. 種別の判定（Type A / B / C）

```
使わせる？ ── yes → Type B（利用保証型）
   └ no（読ませる）
      証拠はコード？数字？ ── コード → Type A（実証型）
                            └ 数字 → Type C（実験型・Lab）
```

| | **Type A: 実証型** | **Type C: 実験型（Lab）** | **Type B: 利用保証型** |
|---|---|---|---|
| 性格 | 「やってみた／移行を実践した」サンプル・ポートフォリオ | 問いと実測結果が本体の実験リポ。配布は追試経路として一応ある、という位置 | 第三者が導入して使えると保証されたツール／ライブラリ |
| 主張の型 | 「**作れた**」— この構成・この移行は成立する | 「**分かった**」— 測ったらこうだった | 「**使える**」— 何ができ、どう導入し、何を保証するか |
| 読者（ターゲットの既定） | レビュアー・採用担当・技術判断者。**コードと設計判断を読む人** | 結果と方法を読む人。追試したい人は少数派の第二読者 | 利用者・導入者。**動かして組み込む人** |
| 結論をいつ知ったか | 着手時に既知（成立させることが目的） | 実行するまで未知（知ることが目的） | —（結論ではなく保証を提供する） |
| 再現性の位置づけ | 不要（見れば分かる） | **生命線**（シード・素材固定が主張の担保） | 保証の一部（環境を問わず動く） |
| 構成の論理 | ナラティブ（課題 → 解決 → 判断の根拠） | 論文の縮約形（問い → 方法 → **結果** → 考察 → 追試） | 利用導線（導入 → 使い方 → 設定 → 制約） |

**A / C のリトマス試験**（迷ったらこちらで決める）:

1. **結果の数字を全部消したら、リポの価値は残るか。** 残るなら A（コードが本体）、残らないなら C（数字が本体）。
2. **結論は作り始める前から分かっていたか。** 分かっていた（できることを示す）なら A、測るまで分からなかったなら C。

## 2. 下限（必ず満たす観点）

種別ごとに、README が満たすべき観点を定める。**これは順序なしのチェックリストであり、H2 の構成そのものではない**。すべて満たしていれば、H2 をどう切り・どう並べ・どう命名するかは自由。

### Type A の下限

- コアメッセージが H1 直下に1文で置かれている
- 動かす手順がある（実証の入口・コピペで動く状態）
- 実証の本体が「課題 → 解決」の論理で語られている
- 技術選定に Why がある（何を使うかだけでなく、なぜそれか）
- スコープが明示されている（何に特化し／何をやらないか）

### Type B の下限

- コアメッセージが H1 直下に1文で置かれている
- 導入手順がある（Installation / Quick Start）
- 最小の使用例がある（Usage）
- 設定・API など、使うために必要な参照情報がある
- スコープが明示されている（対応範囲／非対応）

### Type C の下限

- コアメッセージ（**問い**）が H1 直下に1文で置かれている
- **結果が README 本文にある**（表・数値。リンク先送りにしない。ここが A / B との最大の違い）
- 方法の要約がある（全文は docs/ へ退避してよい）
- 追試手順がある。保証の言い方は B と逆:「同じ手順・同じシード・同じ素材版で同じ数字が出ることは設計している。それ以外の環境・用途は保証しない」
- スコープが明示されている（何を測り、何を測らないか）

Type C に不要なもの: 課題→解決ナラティブ、技術選定の Why の本文掲載（測定装置の選定理由は docs/ 行きで足りる）。

> **筋が通っているかの判定基準**：コアメッセージ文 → H1直下概要 → 各 H2 が、一直線に同じ主張を支えているか。途中で別の話に逸れていたら、その H2 はコアメッセージかターゲットのどちらかとズレている。

## 3. コアメッセージの決め方

> この README を見た人に、ひとことで何だと思ってほしいか。

下限を満たす前提で、リポごとの「伝えたいこと」をここで1文に固定する。これが README 全体の背骨になる。

- **1文**で書く。主語は「このリポは」（または相当する主部）で始める
- 「**何を解決する／実証する／問う**」＋「**どういう手段で**」を含める
- 自慢の形容詞（"高速""モダン""高品質"）と機能の単純列挙は禁止
- **この1文を、そのまま H1 直下の概要1〜2行に一致させる**（コアメッセージ＝概要）

例（Type A）：「レガシーな WinForms 業務アプリを `.NET 8 + React` へ段階的に移行し、AI 自然言語インターフェース追加までの解体・再構成プロセスを実践するサンプル」

例（Type C）：「冒頭の定義文を消された Wikipedia 記事がどの記事名に対応するかを、係数固定の数式・再学習する GBDT・LLM 再判定の組み合わせで解き、コーパスを引き直しても当て続けられるかを実測するベンチマーク」

---

## 4. ターゲットの決め方

開発者である前提で、誰に向けるかを1つ選ぶ。ターゲットは下限を変えない。**下限の上で「何を厚く・何を先頭に・どう命名するか」を決めるレバー**として働く。

| 軸 | 例 | 構成への効き方 |
|---|---|---|
| 習熟度 | ジュニア／ミドル／シニア | Quick Start の手取り足取り度、前提の省略可否 |
| 職種・レイヤー | フロント／バック／インフラ／特定ドメイン担当 | Architecture と Tech Stack のどちらを厚くするか |
| 技術利用者 | 「この技術を使う人」向けか「成果物を使う人」向けか | 利用例・API を出すか、設計思想を出すか |

ターゲットが変われば、同じ素材でも厚みと順序が変わる。コアメッセージとターゲットの2つが、次のアウトラインを導出する。

## 5. アウトラインの組み立て

下限（§2）を満たすことを最低条件に、コアメッセージ（§3）とターゲット（§4）から H2 を動的に決める。**H2 の順序・命名・分割は固定しない**。

下限を満たした上で、必要に応じて次の「素材プール」から H2 を選んで足す。

| 素材 | 内容 | 向く種別 |
|---|---|---|
| **Overview** | 目的・背景・デモ URL | A / B / C |
| **Architecture** | 構成図（Mermaid）・データフロー・レイヤー構成。文章より図 | A / B |
| **Before / After**（命名自由） | 移行前の課題と移行後の構造。実証の本体 | A |
| **Results** | 実測結果の表・グラフと、その読み方。実験の本体 | C |
| **Method** | タスク定義・測定手順の要約（全文は docs/ へ） | C |
| **Tech Stack** | 表形式。技術名＋選定理由（Reason 列） | A / B |
| **Design Decisions** | JUDGE.md から統合した選定根拠。Why を書く | A / B / C |
| **Usage / API** | 使用例・エンドポイント・設定 | B |
| **Reproduce** | 追試手順（シード・素材版の固定、必要リソースの注意） | C |
| **Scope** | Focus（何に特化）と Out-of-Scope（何をやらない） | A / B / C |
| **Deploy** | デプロイ方式の概要のみ（§8 参照） | 公開アプリのみ |
| **Comparison** | 姉妹リポ・既存手法との比較 | A / C |
| **Directory Structure** | ツリー形式。主要ファイルにコメント | A / B |

### H1 + バッジ

```markdown
# Project Name

[![CI](https://github.com/{owner}/{repo}/actions/workflows/ci.yml/badge.svg)](...)
[![Deploy](https://github.com/{owner}/{repo}/actions/workflows/deploy.yml/badge.svg)](...)

（コアメッセージと一致する1〜2行の概要）
```

Deploy バッジは自動デプロイがあるリポのみ。

### 動かす手順（Quick Start / Installation）

Docker で動くなら Docker だけの手順を先に書く。言語ランタイムの個別インストールは Local Development セクションへ分ける。コピペで動く状態にする。

```markdown
## Quick Start

### Prerequisites
- [Docker Desktop](https://www.docker.com/products/docker-desktop/)

### Setup
\```bash
cp .env.example .env
docker compose up -d --build
\```

- App: http://localhost:{port}
```

Type C ではこの位置づけが「追試手順」になる。大きな素材の取得（数GBのダンプ等）が要る場合は、その旨とサイズ・取得先を明記し、結果だけ読む読者が踏まなくて済む導線にする（生成済みデータの Releases 添付など）。

### Tech Stack

技術名と選定理由を表で書く。「何を使っているか」だけでなく「なぜそれか」まで。

```markdown
## Tech Stack

| Layer | Technology | Reason |
|---|---|---|
| Frontend | React, TypeScript, Vite | ... |
| Backend | .NET 8 (Minimal API) | ... |
```

## 7. JUDGE.md / PLAN.md の統合と docs/ 分離

### JUDGE.md / PLAN.md の統合（統合 = 移設 + 削除）

開発中、技術選定の判断基準を `JUDGE.md` に、MVP の定義・完成条件を `PLAN.md` に記録する。これらは AI への指示ファイルではなく、user の作業産物。公開時に README へ統合し、**統合したら元ファイルは削除する**（履歴は git にある。README と原本の二重管理を残さない）。

- `JUDGE.md` … 要点を README の Design Decisions と Tech Stack の Reason 列へ。全文は `docs/design-decisions.md` へ移設（分量が少なければ README に直接統合し docs/ は作らない）→ 削除
- `PLAN.md` … 公開時点で MVP は完了しており役目を終えている。生きている内容（Scope に相当するもの）だけ README の Scope へ吸収。Type C ではタスク定義・生成規模など方法の記述を `docs/method.md` へ移設 → 削除
- **削除前の必須手順**: リポ内の参照を grep する（CLAUDE.md・context/・issues/ が指していることがある）。参照先を README / docs/ へ書き換えてから消す

> AI への前提：「`JUDGE.md` があれば、その判断基準を README の選定理由へ反映する」。判断基準そのものを AI が創作しない。

### docs/ 分離

分離は目的ではなく、**README を1本の主張に保つための退避先**。書く前に分離を前提にせず、README に書いてみて肥大した節だけ退避する。

**分離する条件（いずれか）**:
- その節の全文が、README のコアメッセージの流れと違う読者モーメント（深掘り・参照・運用）に仕える
- README には「導入判断に効く要点」だけあれば足り、全文は読み飛ばされる分量になっている

**分離しない条件**:
- 全文でも数行〜十数行で、分離しても README がほぼ縮まない
- README 側の要約と原本がほぼ同文になる（二重管理だけが残る）

**典型候補**（閉じたリストではない。該当するものだけ作り、空ファイルを揃えない）:

| ファイル | 中身 | 対象リポ |
|---|---|---|
| `docs/design-decisions.md` | 判断の全文（何を捨てたか・再検討の境界） | JUDGE.md があるリポ全般 |
| `docs/method.md` | タスク定義・コーパス生成・測定手順の全文 | Type C |
| `docs/reproduce.md` | 追試手順の全文（README 内に収まらない場合のみ） | Type C |
| `docs/usage.md` | API 表・環境変数・CLI 詳細 | Type B / アプリ系 |
| `docs/release.md` | リリース・publish 手順 | レジストリ公開リポのみ |
| `docs/deploy.md` | 利用者向けセルフホスト手順 | clone リファレンス型アプリ（Secrets を含む自分用運用手順は従来通り README 外・非公開管轄） |

分離したら README 側には要点 + `docs/` へのリンクを残す。`docs/` は README 用アセット（demo GIF・録画スクリプト）の置き場も兼ねる。

**言語対**: `README.en.md` があるリポでは、分離した docs も `docs/*.en.md` を対で作り、両ファイル冒頭に言語リンクを置く（英語版 README だけ docs/ 相当を本文にインラインで持つ、という非対称を作らない）。

**Type C の結果は分離しない**: 結果の表と考察は README 本文が定位置（§2 下限）。docs/ へ送ってよいのは方法の全文・生データ・追加のグラフまで。

## 8. 図と Deploy セクション

### 図

構成図・データフローは Mermaid で書く。README 内にインラインで埋め込み、GitHub 上でレンダリングさせる。

画像を使う場合は `src/` や `docs/` に置き、相対パスで参照する。外部ホスティング（imgur 等）に依存しない。

### Deploy

README にはデプロイ方式の概要と、デモ URL があればそれだけを書く。

```markdown
## Deploy

**Demo:** https://{subdomain}.{domain}

main ブランチへの push で GitHub Actions が自動デプロイ（`cicd-guide.md` compose 型）。
Cloudflare Tunnel 経由で公開。
```

Secrets 一覧・初回セットアップ手順・サーバー側設定は `cicd-guide.md` と運用ドキュメントの責務。README に重複させない。
